The story behind the switchboard

Why I built this

I kept switching browser tabs to compare GPT, Claude, Gemini, and DeepSeek on the same question. That got old fast. So I built Switchboard — one input, every model, responses side by side.

How it works

1

Add lines

Each "line" is a model slot. Choose a provider, type the exact model ID from the provider's docs, and give it a nickname if you like.

2

Set your API keys

Keys go into your browser's localStorage — they never touch this server. Each request carries the key directly from your browser to the provider's API.

3

Type and send

Your prompt is broadcast to all active lines in parallel. Responses stream back as each model finishes — fastest first, no blocking.

4

Compare and continue

Each line keeps its own conversation thread. Flip lines on or off anytime — mute a model mid-chat without losing your place.

Privacy & security

🔐
API keys never leave your browser

Keys are stored only in localStorage on your device. The backend receives them per-request only to forward to the provider — they are never logged or stored server-side.

🚫
No accounts, no tracking

Switchboard has no user accounts, no analytics, and no database of your conversations. Everything lives in your browser's local storage.

🛡️
Strict security headers

Every response includes X-Content-Type-Options, X-Frame-Options: DENY, Referrer-Policy, and a tight Content-Security-Policy. No third-party scripts load at runtime.

✅
Input validation

Prompts are capped at 32 000 characters, model IDs are sanitised by regex, and history messages are trimmed and validated before any outgoing request is made.
